# Формы и JSON в SG-Panel

SG-Panel использует одну модель конфигурации в SQLite. Графическая форма и JSON не являются отдельными режимами: они редактируют один и тот же объект.

## Для кого это нужно

Большинство пользователей продолжает работать через обычные формы. JSON нужен тем, кто хочет использовать параметры Xray, которых пока нет в интерфейсе, например:

- дополнительные `streamSettings`;
- `sockopt`;
- `mux`;
- расширенные поля routing rule;
- `routing.balancers`;
- новые параметры свежих версий Xray.

## Выходы

На странице **Outbounds** доступны два варианта:

- **«Добавить выход»** — обычная форма;
- **«Создать из JSON»** — полный объект outbound.

Для существующего выхода доступны кнопки **«Форма»** и **«JSON»**.

Пример:

```json
{
  "_sgPanel": {
    "name": "Европейский сервер",
    "enabled": true
  },
  "tag": "eu-exit",
  "protocol": "vless",
  "settings": {
    "address": "eu.example.com",
    "port": 443,
    "id": "00000000-0000-4000-8000-000000000000",
    "encryption": "none",
    "flow": "xtls-rprx-vision"
  },
  "streamSettings": {
    "network": "raw",
    "security": "reality",
    "realitySettings": {
      "serverName": "www.bing.com",
      "fingerprint": "chrome",
      "password": "PUBLIC_KEY",
      "shortId": "0123456789abcdef",
      "spiderX": ""
    },
    "sockopt": {
      "tcpFastOpen": true
    }
  }
}
```

Если после этого изменить адрес через форму, `sockopt` останется в объекте.

## Блок `_sgPanel`

Этот блок хранит только данные интерфейса:

- понятное название;
- приоритет;
- состояние объекта.

Перед формированием рабочего `config.json` SG-Panel удаляет `_sgPanel`. Xray его не получает.

## Правила маршрутизации

Для каждого правила доступны:

- обычная форма;
- JSON конкретного rule;
- JSON всего объекта `routing`.

Пример отдельного правила:

```json
{
  "_sgPanel": {
    "name": "Блокировка рекламы",
    "priority": 10,
    "enabled": true
  },
  "type": "field",
  "domain": [
    "geosite:category-ads-all"
  ],
  "outboundTag": "blocked"
}
```

Допускается цель через `balancerTag`:

```json
{
  "_sgPanel": {
    "name": "Европа через пул",
    "priority": 20,
    "enabled": true
  },
  "type": "field",
  "domain": [
    "geosite:fr",
    "geosite:de"
  ],
  "balancerTag": "europe"
}
```

## Полный JSON routing

Полный редактор показывает:

- `domainStrategy`;
- `rules`;
- `balancers`;
- другие поддерживаемые Xray поля routing;
- `_sgPanel.defaultOutboundTag`.

Сохранение полного routing JSON заменяет текущий список правил в SQLite. Перед сохранением панель проверяет:

- структуру JSON;
- уникальность названий;
- существование `outboundTag`;
- существование `balancerTag`;
- корректность доменов, IP, портов и протоколов.

После сохранения изменения сразу видны в конструкторе и автоматически применяются к Xray.

## Безопасность

JSON не применяется напрямую к запущенному Xray. Обычный порядок сохраняется:

1. данные сохраняются в SQLite;
2. SG-Panel формирует итоговый `config.json`;
3. выполняется `xray run -test`;
4. создаётся резервная копия;
5. конфигурация заменяется атомарно;
6. Xray перезапускается;
7. при ошибке выполняется откат.

## Ограничения текущей версии

- JSON выхода поддерживает VLESS;
- JSON всего routing заменяет набор правил, поэтому перед крупным изменением рекомендуется создать резервную копию;
- обновление геоданных через интерфейс пока не реализовано, но установленные `geoip.dat` и `geosite.dat` отображаются на странице маршрутизации.

## Полный config.json

На странице **Xray Config** доступны два уровня работы:

- предварительный просмотр итогового `config.json` и `xray run -test`;
- кнопка **«Редактировать JSON»**, открывающая тот же документ для расширенного редактирования.

В отличие от обычных форм, полный `config.json` намеренно работает в два этапа. Сначала документ синхронизируется с SQLite, затем пользователь отдельно проверяет и применяет итоговую конфигурацию. Это защищает от случайной замены сразу всех разделов.

При сохранении полного JSON SG-Panel синхронизирует обратно в графические страницы:

- управляемый основной VLESS inbound с внутренним тегом `vless-reality-in`;
- активных пользователей и их UUID;
- выбранный профиль inbound, транспорт, REALITY/TLS, XHTTP или gRPC-параметры;
- поддерживаемые VLESS-выходы;
- правила, балансировщики и параметры routing;
- DNS-серверы, hosts и общие настройки DNS;
- log level и статистику.

Секции и поля, которых ещё нет в GUI, не удаляются. Например, можно добавить:

```json
{
  "observatory": {
    "subjectSelector": ["eu-"],
    "probeURL": "https://www.google.com/generate_204"
  }
}
```

Дополнительные inbounds и outbounds также сохраняются. Выход с протоколом, которого ещё нет в форме, отображается в разделе **Outbounds** как объект **JSON** и может использоваться в графической маршрутизации.

Перед изменением базы панель проверяет присланный документ командой `xray run -test`. После синхронизации повторно проверяется уже сформированный панелью результат. При ошибке база автоматически возвращается к предыдущему состоянию.

Сохранение полного `config.json` не перезапускает Xray. После синхронизации вернитесь на **Xray Config**, выполните проверку и нажмите **«Применить конфигурацию»**. Это единственный редактор, где второй шаг остаётся намеренно.

### Метаданные `_sgPanel`

Полный редактор использует служебные блоки `_sgPanel` для данных, которых нет в формате Xray, например публичного адреса сервера, понятного названия выхода, приоритета и состояния правила. Перед проверкой и применением все такие блоки удаляются.
