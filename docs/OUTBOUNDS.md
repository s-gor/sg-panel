# Создание пользовательских выходов в SG-Panel

Пользовательский выход определяет, куда текущий Xray отправит выбранный трафик. Можно создать несколько независимых выходов, например:

```text
eu-exit
usa-exit
home-exit
```

Каждый выход имеет собственный тег, адрес, UUID, транспорт и защиту. Слово «каскад» в интерфейсе не используется: каскад возникает только как итоговая схема, когда первый сервер направляет трафик через пользовательский выход на другой Xray-сервер.

## Что должно быть подготовлено

На удалённом сервере уже должен работать inbound, а у вас должны быть его клиентские параметры:

- адрес или домен;
- порт;
- UUID;
- транспорт;
- тип защиты;
- дополнительные параметры REALITY, TLS или XHTTP.

Проще всего брать значения из рабочей VLESS-ссылки удалённого сервера или из его клиентского JSON.

## Общий порядок

1. Откройте **Outbounds**.
2. Нажмите **«Добавить выход»**.
3. Выберите transport и security.
4. Заполните параметры удалённого Inbound.
5. Нажмите **«Создать выход»**.
6. Дождитесь сообщения, что настройки сохранены и Xray работает.
7. Откройте **Routing** и назначьте новый выход Default Outbound либо используйте его в отдельном правиле.

Создание Outbound сразу добавляет его в рабочий `config.json`, но не направляет через него трафик без настройки Routing.

## Основные поля

| Поле | Что указывать |
|---|---|
| **Тег** | Уникальное техническое имя, например `eu-exit`. Используется в `routing.rules` как `outboundTag`. |
| **Название** | Понятное имя в панели, например `Европейский сервер`. |
| **Адрес** | Домен или IP удалённого Xray-сервера без `https://`. |
| **Порт** | Публичный порт удалённого inbound. Обычно `443`. |
| **UUID** | UUID пользователя на удалённом сервере. |
| **Fingerprint** | Обычно `chrome`, если удалённая конфигурация не требует другого значения. |
| **Server name / SNI** | Имя из клиентской конфигурации удалённого сервера. Для TLS обычно совпадает с доменом сертификата. |

Тег должен быть уникальным. Не используйте системные теги `direct` и `blocked`.

## Вариант 1. VLESS + RAW/TCP + REALITY

Используйте этот вариант, когда удалённый сервер принимает обычный VLESS Reality на TCP.

Пример клиентской ссылки:

```text
vless://UUID@server.example.com:443?security=reality&type=tcp&sni=www.bing.com&fp=chrome&pbk=PUBLIC_KEY&sid=SHORT_ID&flow=xtls-rprx-vision
```

Заполните:

```text
Протокол:          VLESS
Транспорт:         RAW / TCP
Защита:            REALITY
Тег:               eu-reality
Название:          EU Reality
Адрес:             server.example.com
Порт:              443
UUID:              значение перед @
Flow:              xtls-rprx-vision
Server name / SNI: параметр sni
Fingerprint:       параметр fp
Public key:        параметр pbk
Short ID:          параметр sid
Spider X:          значение из конфигурации, обычно / или пусто
```

Если в рабочей клиентской конфигурации нет `flow`, выберите `none`.

## Вариант 2. VLESS + XHTTP + TLS

Используйте этот вариант, когда удалённый XHTTP inbound доступен через Nginx и обычный TLS-сертификат.

Типовая ссылка:

```text
vless://UUID@xhttp.example.com:443?encryption=none&security=tls&type=xhttp&host=xhttp.example.com&path=%2Frandom-path&sni=xhttp.example.com&fp=chrome
```

Заполните:

```text
Протокол:          VLESS
Транспорт:         XHTTP
Защита:            TLS
Тег:               eu-xhttp
Название:          EU XHTTP TLS
Адрес:             xhttp.example.com
Порт:              443
UUID:              значение перед @
Flow:              none (панель установит автоматически)
Host:              параметр host; при отсутствии можно оставить пустым
Path:              декодированный параметр path, например /random-path
Mode:              auto, если сервер не требует явно другой режим
Server name / SNI: параметр sni или домен сертификата
Fingerprint:       параметр fp, обычно chrome
ALPN:               только если он явно указан в рабочей конфигурации
Не проверять сертификат: выключено
```

### Сервер из проекта xray-xhttp-v2

В базовой схеме этого проекта Nginx принимает соединение на `443`, а Xray слушает `127.0.0.1:8443`. В SG-Panel указывайте **публичный домен и порт Nginx**, а не локальный порт `8443`.

Значения берутся из `/root/xray-xhttp-values.env` удалённого сервера:

```text
DOMAIN       -> Адрес, Host и Server name / SNI
USER_UUID    -> UUID
XHTTP_PATH   -> Path
443          -> Порт
```

Для базовой конфигурации выберите `Mode: auto`. Локальный `XHTTP_LOCAL_PORT=8443` в SG-Panel не переносится.

## Вариант 3. VLESS + XHTTP + REALITY

Используйте этот вариант, когда удалённый inbound сочетает XHTTP и REALITY.

Заполните:

```text
Протокол:          VLESS
Транспорт:         XHTTP
Защита:            REALITY
Тег:               eu-xhttp-reality
Название:          EU XHTTP Reality
Адрес:             домен или IP удалённого сервера
Порт:              публичный порт inbound
UUID:              UUID пользователя
Flow:              none
Host:              значение XHTTP host или пусто
Path:              XHTTP path
Mode:              режим из клиентской конфигурации
Server name / SNI: Reality serverName
Fingerprint:       обычно chrome
Public key:        Reality public key/password
Short ID:          Reality shortId
Spider X:          значение из клиентской конфигурации или пусто
```

Все параметры XHTTP и REALITY должны совпадать с рабочей клиентской конфигурацией удалённого сервера.

## Как перенести параметры из VLESS-ссылки

Пример:

```text
vless://UUID@host.example.com:443?security=tls&type=xhttp&host=host.example.com&path=%2Fabc123&sni=host.example.com&fp=chrome
```

Соответствие:

| Часть ссылки | Поле SG-Panel |
|---|---|
| `UUID` перед `@` | UUID |
| имя после `@` | Адрес |
| `:443` | Порт |
| `type=xhttp` | Транспорт XHTTP |
| `security=tls` | Защита TLS |
| `host=` | Host |
| `path=` | Path после URL-декодирования |
| `sni=` | Server name / SNI |
| `fp=` | Fingerprint |
| `pbk=` | Public key / password |
| `sid=` | Short ID |
| `flow=` | Flow |

Не переносите `https://` в поля **Адрес** или **Host**.

## Как направить трафик через выход

### Весь трафик

Откройте **Routing**, выберите новый тег в поле **«Выход по умолчанию»** и нажмите **«Сохранить и применить»**.

### Только выбранные домены или IP

Оставьте выходом по умолчанию `direct` и создайте правило:

```text
Название:   Через EU
Приоритет:  30
Выход:      eu-exit
Домены:     domain:example.com
```

Можно использовать домены, IP/CIDR, порты, сеть, протоколы, inbound tags и конкретных пользователей.

## Проверка

После создания выхода:

1. Нажмите **«Проверить TCP»** на карточке выхода.
2. Убедитесь на странице **Обзор**, что Xray имеет состояние `active`.
3. Настройте Routing.
4. Проверьте реальный выходной IP или доступ к выбранному ресурсу через клиент.

При необходимости откройте **Xray Config** и убедитесь, что объект присутствует в массиве `outbounds` с правильными `tag`, `address`, `port`, `id`, `network` и `security`.

Проверка TCP подтверждает только сетевую доступность адреса и порта. Успешная проверка не доказывает правильность UUID, SNI, Path, ключа REALITY или TLS-сертификата.

## Типичные ошибки

### XHTTP не подключается

Проверьте:

- Path начинается с `/`;
- Path совпадает с удалённым inbound и конфигурацией Nginx;
- Host не содержит `https://` и путь;
- Flow равен `none`;
- выбран правильный Mode;
- публичный порт открыт.

### REALITY не подключается

Проверьте:

- Server name / SNI;
- public key/password;
- Short ID;
- Fingerprint;
- Flow для RAW/TCP.

### TLS не подключается

Проверьте:

- домен указывает на удалённый сервер;
- сертификат действителен для указанного SNI;
- `Не проверять сертификат` выключено;
- Host, Path и порт соответствуют удалённой конфигурации.

## Изменение и удаление

Выход нельзя отключить или удалить, пока он:

- выбран выходом по умолчанию;
- используется активным правилом маршрутизации.

Сначала измените маршрутизацию, затем отключите или удалите выход.

## Редактирование выхода через JSON

Для каждого пользовательского выхода доступны кнопки **«Форма»** и **«JSON»**. Это не два режима, а два представления одного объекта.

Создание и изменение Outbound через JSON также сразу проверяет и применяет конфигурацию Xray.

JSON удобен для параметров, которых пока нет в графической форме, например:

```json
"sockopt": {
  "tcpFastOpen": true,
  "tcpKeepAliveIdle": 60
}
```

или:

```json
"mux": {
  "enabled": false
}
```

Дополнительные поля сохраняются при последующем изменении адреса, порта, UUID, транспорта или защиты через форму.

Блок `_sgPanel` содержит название и состояние выхода и не включается в итоговую конфигурацию Xray.

Подробно: [Формы и JSON](JSON-EDITOR.md).
