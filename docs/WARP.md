# Cloudflare WARP в SG-Panel

SG-Panel создаёт WARP как встроенный WireGuard Outbound Xray с тегом:

```text
warp
```

Системный `warp-cli`, интерфейс `wg-quick@warp` и изменение маршрутов Ubuntu не требуются.

## Где что настраивается

| Задача | Раздел |
|---|---|
| создать, проверить, включить, пересоздать или удалить WARP | **Outbounds** |
| выбрать, какой трафик идёт через WARP | **Routing** |
| увидеть endpoint, Default Outbound и журналы | **Diagnostics** |

## Путь трафика

```text
Клиент — основной Inbound — Xray — Routing — warp — Cloudflare — Интернет
```

WARP не меняет основной Inbound. Он работает с каждым из трёх профилей.

## Создание

Откройте **Outbounds** и нажмите **«Создать WARP»**.

SG-Panel:

1. регистрирует профиль через `wgcf-cli`;
2. создаёт WireGuard Outbound Xray;
3. закрепляет IPv4 endpoint `162.159.192.1:2408`;
4. сохраняет учётные данные;
5. включает `noKernelTun`;
6. применяет обновлённый `config.json`.

Ожидаемое состояние карточки:

```text
WARP настроен
Активен
```

Создание WARP ещё не направляет через него весь трафик.

## Обязательная проверка

До изменения Routing нажмите **«Проверить WARP»**.

SG-Panel временно запускает отдельный локальный SOCKS-вход Xray и выполняет запрос через WARP. Рабочий Inbound не останавливается.

Ожидается:

```text
WARP on, IP ...
```

или:

```text
WARP plus, IP ...
```

IPv6 в результате является нормальным выходным адресом Cloudflare.

Не включайте весь трафик через WARP, пока отдельная проверка не прошла.

## Режимы Routing

### Не использовать WARP

```text
Routing — Не использовать WARP — Сохранить и применить
```

WARP остаётся настроенным, но Default Outbound обычно равен `direct`.

### Только выбранные сайты

```text
Routing — Только выбранные сайты
```

Укажите домены, например:

```text
domain:google.com
domain:spotify.com
domain:reddit.com
geosite:instagram
geosite:meta
```

Остальной трафик продолжит использовать Default Outbound.

### Весь трафик через WARP

```text
Routing — Весь трафик через WARP — Сохранить и применить
```

`warp` становится Default Outbound.

В таблице правил может не появиться отдельная строка WARP. Это нормально: выход выбирается как маршрут по умолчанию.

Явные правила с меньшим приоритетом, например `blocked`, выполняются раньше.

## Проверка результата для клиента

После применения:

1. отключите клиентское подключение;
2. подключитесь снова;
3. проверьте внешний IP;
4. откройте **Обзор** и убедитесь, что Default Outbound равен `WARP`.

IP клиента не должен принадлежать EC2/AWS при режиме всего трафика.

## Отключение

Кнопка **«Отключить»**:

- исключает `warp` из рабочего `config.json`;
- возвращает Default Outbound на `direct`, если это необходимо;
- отключает связанные правила;
- сохраняет учётные данные для последующего включения;
- сразу применяет конфигурацию.

## Пересоздание

Кнопка **«Пересоздать»** регистрирует новый WARP-профиль и заменяет WireGuard-ключи и адреса.

После пересоздания снова выполните отдельную проверку WARP.

Endpoint останется:

```text
162.159.192.1:2408
```

## Удаление

Кнопка **«Удалить»**:

- удаляет Outbound `warp`;
- удаляет связанные правила;
- возвращает Default Outbound на `direct`;
- удаляет сохранённые данные WARP;
- применяет конфигурацию.

## Где хранятся данные

```text
SQLite panel.db
/etc/xpanel-mvp/warp
```

Private key не показывается в интерфейсе. Файлы WARP и резервные копии являются секретными.

## Типичные ошибки

### Проверка зависает и завершается timeout

Проверьте в **Diagnostics**:

```text
WARP endpoint: 162.159.192.1:2408
```

Также проверьте исходящий UDP.

Причиной на EC2 без IPv6-маршрута ранее был доменный endpoint `engage.cloudflareclient.com`, который мог разрешаться в IPv6.

### После включения сайты не открываются

Верните рабочий маршрут:

```text
Routing — Не использовать WARP — Сохранить и применить
```

Затем проверяйте WARP отдельно.

### Клиент показывает IP AWS

Проверьте:

```text
WARP активен
тест WARP прошёл
Routing установлен в режим всего трафика
Обзор показывает Default Outbound WARP
```

### Выборочные домены не работают

Проверьте:

- формат `domain:`, `full:`, `keyword:` или `geosite:`;
- порядок правил;
- наличие sniffing для доменных правил;
- отсутствие более раннего совпавшего правила.
